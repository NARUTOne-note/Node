# Database 数据库

> 数据库基础、主流数据库特性对比与选型、大数据技术体系与数据治理技术梳理。
> 各数据库的详细使用笔记见 `dbs/` 目录（mysql.md / mongodb.md / redis.md）。

## 一、数据库基础

### 基本概念

1. **数据库（DB）**：按照一定的数据模型组织、存储和管理数据的仓库。
2. **数据库管理系统（DBMS）**：管理数据库的软件，如 MySQL、Oracle、Redis，提供数据定义（DDL）、数据操作（DML）、数据控制（DCL）能力。
3. **数据库系统（DBS）**：数据库 + DBMS + 应用程序 + 管理员的统称。

### 数据库发展史

1. **网状数据库 / 层次数据库**：早期以图、树结构组织数据，导航式访问，缺乏独立性。
2. **关系数据库（RDBMS）**：以关系代数为理论基础，用二维表组织数据，使用 SQL 操作，代表产品 Oracle、MySQL、PostgreSQL。
3. **NoSQL 数据库**：为解决互联网海量数据、高并发、灵活 schema 问题而生，放弃部分 ACID 换取高性能与水平扩展。
4. **NewSQL 数据库**：既要 NoSQL 的水平扩展能力，又要关系数据库的 ACID 与 SQL，代表产品 TiDB、CockroachDB、OceanBase。
5. **云原生数据库**：存算分离、Serverless、弹性伸缩，如 AWS Aurora、PolarDB、Snowflake。

### ACID、CAP 与 BASE

**ACID**（关系数据库事务的四大特性）：

- **原子性（Atomicity）**：事务中的操作要么全部成功，要么全部失败回滚。
- **一致性（Consistency）**：事务执行前后，数据都满足完整性约束。
- **隔离性（Isolation）**：并发事务之间互不干扰，通过隔离级别（读未提交 / 读已提交 / 可重复读 / 串行化）在一致性与并发性能间权衡。
- **持久性（Durability）**：事务提交后，数据永久保存，掉电不丢失。

**CAP 定理**：分布式系统中，一致性（Consistency）、可用性（Availability）、分区容错性（Partition tolerance）三者最多同时满足两个。网络分区无法避免，所以实际是在 P 的前提下做 **CP**（如 HBase、ZooKeeper）或 **AP**（如 Cassandra、Eureka）的取舍。

**BASE 理论**：对 CAP 中 AP 方案的延伸，是 NoSQL 的设计哲学。

- **Basically Available**：基本可用，允许损失部分可用性（响应时间、功能降级）。
- **Soft state**：软状态，允许数据存在中间状态。
- **Eventually consistent**：最终一致性，不要求实时一致，但保证经过一段时间后达到一致。

## 二、关系型数据库（RDBMS）

### 共同特性

- 二维表结构，schema 固定（schema-on-write），支持 SQL 标准。
- 支持 ACID 事务、索引、视图、存储过程、约束，数据一致性强。
- 单机性能强，但水平扩展困难（通常靠分库分表、读写分离、中间件）。

### 主流产品对比

| 数据库 | 特性 | 适用场景 |
| --- | --- | --- |
| **MySQL** | 开源免费、社区活跃、生态最丰富；InnoDB 引擎支持事务、行级锁、MVCC；主从复制成熟；功能相对精简 | 互联网业务、中小规模 Web 应用，国内事实上的默认选择 |
| **PostgreSQL** | 开源中功能最强：JSONB、数组、自定义类型/函数、窗口函数、CTE、PostGIS 地理扩展；SQL 标准支持最全 | 复杂查询、GIS 地理信息、对数据类型与扩展性要求高的业务、SaaS 多租户 |
| **Oracle** | 商业数据库，功能全面、稳定可靠；RAC 集群、Data Guard 容灾；配套工具与服务体系完善 | 银行、电信、政务等核心业务，预算充足、对稳定性要求极致的场景 |
| **SQL Server** | 微软系，与 Windows / .NET / Azure 深度集成；图形化管理友好 | 微软技术栈企业、Windows Server 环境 |
| **SQLite** | 嵌入式、单文件、零配置、无服务端进程；不支持高并发写 | 移动端 App、IoT 设备、桌面软件、本地缓存、测试环境 |

### 关系型数据库的适用边界

1. 适合：强一致事务（转账、订单、库存）、结构化数据、复杂关联查询。
2. 不适合：海量数据水平扩展（单机瓶颈）、超高并发 KV 读写（应上缓存）、非结构化/半结构化数据、灵活多变的 schema。

## 三、NoSQL 数据库

NoSQL（Not Only SQL）按数据模型分为几大类，各自解决不同的问题。

### 键值数据库（Key-Value）

| 数据库 | 解决的问题 | 特性 | 适用场景 |
| --- | --- | --- | --- |
| **Redis** | 磁盘 DB 扛不住的高并发读写、热点数据访问 | 内存存储、单线程事件循环（6.0 起 I/O 多线程）；支持 String/Hash/List/Set/ZSet/Bitmap/GEO/Stream 等丰富结构；RDB/AOF 持久化；主从、哨兵、Cluster 集群；支持 Lua、过期、发布订阅 | 缓存、会话共享、排行榜、计数器、分布式锁、限流、轻量消息队列 |
| **Memcached** | 纯缓存场景 | 简单 KV、多线程、LRU 淘汰、无持久化、无主从 | 只需简单缓存、不需要持久化和复杂结构的场景（现多被 Redis 替代） |

### 文档数据库（Document）

| 数据库 | 解决的问题 | 特性 | 适用场景 |
| --- | --- | --- | --- |
| **MongoDB** | 关系库 schema 僵化、迭代慢；半结构化数据存储 | BSON 文档、schema 灵活；二级索引、聚合管道；副本集 + 分片水平扩展；4.0 起支持多文档事务 | 内容管理（文章、商品）、快速迭代的业务、日志/行为记录、JSON 风格 API 后端 |
| **CouchDB** | 多端数据同步 | 多版本并发控制（MVCC）、双向复制 | 离线优先应用、多端同步 |

### 列族 / 宽列数据库（Wide-Column）

| 数据库 | 解决的问题 | 特性 | 适用场景 |
| --- | --- | --- | --- |
| **HBase** | 单表百亿/千亿行级别的海量数据随机读写 | 基于 HDFS、列族存储、稀疏表；强一致（单行）、按 RowKey 范围分区自动分裂；依赖 HDFS + ZooKeeper，运维重 | 用户画像、风控特征、消息/订单明细的海量存储、时序类数据 |
| **Cassandra** | 多机房部署下的高可用写入 | 去中心化 P2P 架构、无单点；AP 模型、可调一致性；CQL 类 SQL | 写入密集、跨地域多活、写多读少（消息记录、物联网上报） |

### 图数据库（Graph）

| 数据库 | 解决的问题 | 特性 | 适用场景 |
| --- | --- | --- | --- |
| **Neo4j** | 关系库做多跳关联查询（JOIN 爆炸）性能极差 | 原生图存储（节点、边、属性）；Cypher 图查询语言；单机为主 | 社交关系、风控反欺诈（关联团伙）、知识图谱、推荐、血缘分析 |
| **NebulaGraph** | 超大规模图的分布式存储 | 分布式、shared-nothing、国产开源 | 十亿级以上点边的超大规模图场景 |

### 时序数据库（TSDB）

| 数据库 | 解决的问题 | 特性 | 适用场景 |
| --- | --- | --- | --- |
| **InfluxDB** | 按时间持续写入的指标数据，通用库写入与聚合查询效率低 | 时间戳为主键、高压缩、TTL 自动过期、连续聚合 | 监控指标、传感器数据 |
| **Prometheus** | 云原生环境监控告警 | Pull 模型采集、PromQL、与 K8s/Grafana 生态一体 | 服务与容器监控告警体系 |
| **TDengine** | 物联网海量设备时序数据 | 「一个设备一张表」+ 超级表模型、高压缩、国产开源 | 车联网、工业物联网 |

### 搜索引擎（Search Engine）

| 数据库 | 解决的问题 | 特性 | 适用场景 |
| --- | --- | --- | --- |
| **Elasticsearch** | 关系库 LIKE 无法支撑全文检索与复杂日志分析 | 基于 Lucene、倒排索引、近实时（NRT）；RESTful API；分词、相关性打分；ES+Logstash+Kibana 即 ELK | 全文搜索、日志采集分析（ELK）、聚合分析、地理检索 |
| **Solr / OpenSearch** | 同上 | Solr 老牌稳定；OpenSearch 是 AWS 对 ES 的开源分支 | 已有对应技术栈的团队 |

## 四、NewSQL / 分布式数据库

> 解决的问题：关系库单机扩展到顶后，分库分表中间件方案侵入业务、跨分片事务与 JOIN 困难。NewSQL 目标是「像用单机 MySQL 一样用分布式数据库」。

| 数据库 | 特性 | 适用场景 |
| --- | --- | --- |
| **TiDB** | 兼容 MySQL 协议；TiKV（行存）+ TiFlash（列存）实现 HTAP；Raft 多副本、自动扩缩容 | MySQL 到顶后的平滑替换、既要交易又要实时分析（HTAP） |
| **OceanBase** | 蚂蚁自研，金融级高可用（RPO=0）；兼容 MySQL/Oracle 模式；单机分布式一体化 | 金融核心、对数据安全与容灾要求极高的场景 |
| **CockroachDB** | 兼容 PostgreSQL 协议；全球部署、多活、强一致 | 跨地域全球化业务 |
| **Google Spanner** | TrueTime（原子钟+GPS）实现全球外部一致性 | 谷歌系全球业务，业界分布式数据库的标杆参考 |

## 五、分析型数据库（OLAP）

> 解决的问题：关系库（OLTP）擅长高频小事务，但对海量数据的多维聚合分析力不从心；Hadoop 系引擎（Hive/Spark）算得动但延迟高。分析型数据库用「列式存储 + MPP 并行计算 + 向量化执行」实现海量数据的秒级甚至亚秒级分析，常直接作为实时数仓的查询服务层。

**Doris 就属于这一类**：它既不是传统关系型 OLTP 库，也不是 NoSQL，而是**基于 MPP 架构的列式分析型数据库（OLAP）**。兼容 MySQL 协议、支持标准 SQL，长得像个「MySQL 集群」，但面向的是分析场景而非交易场景。

| 数据库 | 特性 | 适用场景 |
| --- | --- | --- |
| **Apache Doris** | MPP 架构（FE 管元数据与调度、BE 管存储计算）、列式存储、全向量化执行；兼容 MySQL 协议，接入简单；支持高并发点查、复杂 JOIN、物化视图；明细与聚合模型一体，支持事务性批量导入 | 实时数仓服务层、BI 报表、用户画像、日志分析，国内企业实时分析的主流选择 |
| **ClickHouse** | 列存 + 向量化，单机性能极强；MergeTree 引擎族；多表 JOIN 与高频更新较弱 | 宽表为主的极速分析、日志/行为分析 |
| **StarRocks** | 与 Doris 同源思路，CBO 优化器、向量化、支持数据湖外表联邦查询 | 实时分析、湖仓加速 |
| **Greenplum** | 基于 PostgreSQL 的 MPP 数仓，支持完整事务与存储过程 | 传统企业级数据仓库（离线/准实时） |

**Doris 快速定位**：

1. 从数据模型看：用二维表、标准 SQL，像关系库。
2. 从架构看：shared-nothing MPP 集群，节点水平扩展。
3. 从场景看：面向 OLAP（多维聚合、高并发报表分析），不用于订单、支付这类高频小事务的 OLTP 场景。

## 六、数据库对比与选型

### 各类型总览

| 类型 | 数据模型 | 一致性 | 扩展性 | 典型代表 | 一句话定位 |
| --- | --- | --- | --- | --- | --- |
| 关系型 | 二维表 | 强一致 ACID | 垂直为主 | MySQL、PostgreSQL | 通用业务与事务的基石 |
| 键值 | KV | 最终/单节点强 | 水平极佳 | Redis | 高速缓存与简单热数据 |
| 文档 | JSON 文档 | 单文档强一致 | 水平好 | MongoDB | 灵活 schema 的业务数据 |
| 列族 | 稀疏宽表 | 单行强/可调 | 水平极佳 | HBase、Cassandra | 海量明细随机读写 |
| 图 | 节点+边 | 强一致 | 一般 | Neo4j | 多跳关联关系分析 |
| 时序 | 时间序列 | 最终 | 水平好 | InfluxDB、TDengine | 监控与物联网指标 |
| 搜索 | 倒排索引 | 最终（NRT） | 水平好 | Elasticsearch | 全文检索与日志分析 |
| NewSQL | 二维表 | 分布式强一致 | 水平极佳 | TiDB、OceanBase | 分布式的关系库 |
| 分析型（OLAP） | 列式宽表 | 批量原子导入 | 水平极佳 | Doris、ClickHouse | 海量数据的多维分析与报表 |


为了更好地理解，可以看看在“数据库家族”中的位置：

| 分类  | 代表产品 | 主要用途 |
| --- | --- | --- |
| **关系型数据库 (OLTP)** | MySQL, PostgreSQL | 处理日常事务，如订单、用户信息，强调数据一致性。 |
| **NoSQL 数据库** | Redis, MongoDB | 应对特定场景，如缓存、灵活的数据结构。 |
| **OLAP 数据库** | **Apache Doris**, ClickHouse | **海量数据的快速分析**，如生成报表、支持业务决策。 |
| **数据仓库** | Snowflake, Hive | 存储和管理大量历史数据，用于分析和报告。 |

### 选型原则

1. **先定场景**：OLTP（在线交易）选关系库/NewSQL；OLAP（在线分析）选列式数仓/OLAP 引擎；缓存选 Redis。
2. **数据模型匹配**：关系明确且多变选图库；JSON 风格选文档库；时间维度选时序库；全文检索选搜索引擎。
3. **一致性要求**：钱、库存类必须强一致；日志、统计类可接受最终一致。
4. **数据量与增长**：TB 级以内单机关系库足够；持续海量增长直接考虑分布式方案。
5. **读写比例与延迟**：读多写少可加缓存；写密集考虑 Cassandra/Kafka 削峰。
6. **团队与生态**：优先选团队熟悉、社区活跃、云服务支持好的产品，运维成本往往大于性能差异。

## 七、大数据技术体系

### 大数据的特征（4V）

1. **Volume**：数据量大，TB/PB 级起步。
2. **Velocity**：产生与处理速度快，要求实时/准实时。
3. **Variety**：类型多样，结构化、半结构化（JSON/日志）、非结构化（音视频）。
4. **Value**：价值密度低，需要计算挖掘。

### 总体架构（经典 Lambda 分层）

```text
数据源（业务库/日志/埋点/IoT）
   │  采集：Flume / DataX / CDC / Kafka
   ▼
存储层（HDFS / 对象存储 / HBase / 数据湖格式）
   │  计算：MapReduce / Spark（批） + Flink（流）
   ▼
数仓分层（ODS → DWD → DWS → ADS）
   │  查询：Hive / Presto / ClickHouse / Doris
   ▼
应用层（报表 BI / 推荐 / 风控 / 用户画像）
调度：DolphinScheduler / Airflow；治理：Atlas / Ranger / Griffin
```

### 1. 存储层

| 技术 | 解决的问题 | 特性 | 适用场景 |
| --- | --- | --- | --- |
| **HDFS** | 单机文件系统存不下、不可靠 | 分布式文件系统，NameNode 管元数据、DataNode 存数据块（多副本）；一次写入多次读取、高吞吐；不适合低延迟和海量小文件 | 离线数仓的底层存储 |
| **对象存储（S3/OSS/MinIO）** | 海量非结构化数据的廉价存储 | 按桶+对象组织、几乎无限扩容、按量付费；是云时代**存算分离**的基础 | 图片视频、备份归档、云上数据湖底座 |
| **Kudu** | HDFS 不适合随机写、HBase 分析慢 | 列存 + 支持更新，介于两者之间 | 既要随机更新又要较快分析的中间场景 |

**数据仓库 vs 数据湖 vs 湖仓一体**：

- **数据仓库**：schema-on-write，先建模后入库，数据规范、查询快，但不灵活；代表 Hive、Teradata。
- **数据湖**：schema-on-read，原始数据先进湖（HDFS/S3），用时再解释；灵活便宜，但缺乏管理易变成「数据沼泽」。
- **湖仓一体**：在数据湖之上提供数仓级的 ACID、schema 管理、性能优化，是当前主流方向。

**数据湖表格式**（解决数据湖上的 ACID、schema 演进、时间旅行问题）：

| 技术 | 特性 | 适用场景 |
| --- | --- | --- |
| **Apache Iceberg** | 引擎中立（Spark/Flink/Trino 都支持）、隐藏分区、快照隔离 | 通用湖仓底座，社区势头最强 |
| **Apache Hudi** | 擅长 upsert 与增量处理、CDC 入湖 | 数据库变更同步入湖、近实时湖表 |
| **Delta Lake** | Databricks 出品，与 Spark 深度集成 | Databricks/Spark 技术栈 |

### 2. 计算引擎

| 技术 | 解决的问题 | 特性 | 适用场景 |
| --- | --- | --- | --- |
| **MapReduce** | 第一代海量数据分布式批处理 | 分而治之（Map/Shuffle/Reduce）；中间结果落盘，慢；编程繁琐 | 历史意义大于实用，已被 Spark 取代，理解原理即可 |
| **Spark** | 统一的大规模数据处理 | 内存计算（比 MR 快 10~100 倍）；RDD/DataFrame API；Spark SQL、Streaming（微批）、MLlib 一体化生态 | 离线批处理、ETL、机器学习特征工程，大数据「通用引擎」 |
| **Flink** | 真正的低延迟流处理 | 事件驱动纯流模型；Event Time + Watermark 处理乱序；状态管理与 Checkpoint（异步屏障快照）实现精确一次；流批一体 | 实时数仓、实时风控、实时监控告警、实时推荐 |
| **Storm** | 早期实时计算 | 纯流、延迟低；默认 at-least-once | 基本被 Flink 取代，存量维护 |

### 3. OLAP 查询引擎

| 技术 | 解决的问题 | 特性 | 适用场景 |
| --- | --- | --- | --- |
| **Hive** | 让不会写 MR 的人用 SQL 分析 HDFS 数据 | HQL 翻译为 MR/Tez/Spark 执行；T+1 离线、延迟高； metastore 管理元数据 | 离线数仓事实标准，T+1 批处理 |
| **Presto / Trino** | 交互式（秒级）跨数据源查询 | MPP 内存计算、联邦查询（同时查 Hive/MySQL/ES）；不擅长超长 ETL（容错弱） | 即席查询、数据探索、跨源分析 |
| **ClickHouse** | 单机/集群级的极速分析 | 列式存储、向量化执行、稀疏索引、MergeTree 引擎族；不擅高频更新与多表 JOIN | 实时报表、日志分析、用户行为分析，宽表查询 |
| **Doris / StarRocks** | 国产 MPP OLAP，兼顾易用与性能 | 兼容 MySQL 协议；支持高并发点查与 JOIN；物化视图 | 实时数仓服务层、BI 报表、高并发分析 |

### 4. 资源调度

| 技术 | 解决的问题 | 特性 | 适用场景 |
| --- | --- | --- | --- |
| **YARN** | Hadoop 集群的 CPU/内存资源统一分配 | ResourceManager + NodeManager；队列、配额 | Hadoop 生态（MR/Spark on YARN） |
| **Kubernetes** | 云原生时代的通用容器编排 | 声明式 API、弹性伸缩、自愈；大数据组件逐步迁移（Spark on K8s） | 新建平台优先，统一大数据与业务服务的基础设施 |

### 5. 数据采集与同步

| 技术 | 解决的问题 | 特性 | 适用场景 |
| --- | --- | --- | --- |
| **Flume** | 日志文件采集到 HDFS/Kafka | Source-Channel-Sink 管道、可靠 | 服务器日志汇聚 |
| **Filebeat / Logstash** | 轻量采集 / 结构化处理 | Filebeat 轻量转发，Logstash 重清洗（ELK 一环） | 日志进 Elasticsearch |
| **DataX** | 异构数据源之间的离线批量同步 | 阿里开源、插件式（MySQL/Oracle/HDFS/Hive 等几十种 Reader/Writer） | 业务库 → 数仓的 T+1 全量/增量同步 |
| **Sqoop** | RDBMS ↔ Hadoop 批量迁移 | 命令行工具，已退役（Apache Attic） | 存量老系统，新项目用 DataX/Spark 替代 |
| **Kafka** | 高吞吐消息管道与流数据中枢 | 分布式分区日志、持久化、消费组、削峰填谷；2.8+ KRaft 去 ZooKeeper | 日志/埋点传输、系统解耦、流计算的 Source/Sink |
| **Pulsar** | Kafka 的存算分离替代者 | 计算与存储分层、多租户、跨地域复制 | 多租户消息平台 |
| **RocketMQ** | 业务消息 | 事务消息、顺序消息、延迟消息 | 订单、支付等强业务语义消息 |
| **Canal** | MySQL 增量数据订阅 | 伪装成 MySQL slave 解析 binlog | MySQL → Kafka 的 CDC 同步、缓存失效 |
| **Debezium** | 多数据库通用 CDC | 基于 Kafka Connect，支持 MySQL/PG/Oracle/MongoDB | 异构库的变更捕获 |
| **Flink CDC** | CDC 数据直接进流计算 | 免 Kafka 中间层、全量+增量一体 | 实时数仓 ODS 层建设 |

## 八、数据治理

> 解决的问题：数据多了之后「找不到、看不懂、不敢用、管不住」——口径不一致、质量参差、血缘不清、权限混乱。数据治理是对数据全生命周期的管理体系，目标是**数据可发现、可理解、可信任、可管控**。

### 治理的核心内容

1. **元数据管理**：技术元数据（表结构、分区）、业务元数据（口径、负责人）、管理元数据（权限、生命周期）。
2. **数据血缘**：字段级的来源与去向追踪，支撑影响分析（改一张表会影响哪些报表）与问题定位。
3. **数据质量**：完整性、准确性、一致性、及时性、唯一性、有效性的规则校验与监控告警。
4. **数据标准**：统一命名规范、指标口径（如「活跃用户」全公司一个定义）、数据字典。
5. **数据安全**：分级分类、脱敏、权限管控、审计。
6. **生命周期管理**：冷热分层存储、归档、到期销毁，控制存储成本。
7. **数仓建模规范**：ODS（原始）→ DWD（明细清洗）→ DWS（主题汇总）→ ADS（应用）分层，配合维度建模（事实表 + 维度表），是治理落地的工程载体。

### 治理相关技术

| 技术 | 解决的问题 | 特性 | 适用场景 |
| --- | --- | --- | --- |
| **Apache Atlas** | Hadoop 生态元数据与血缘管理 | Hive/HBase/Kafka 等组件自动采集 hook、血缘图、分类打标 | 传统 Hadoop 数仓的元数据中心 |
| **DataHub** | 现代化元数据平台 | LinkedIn 开源；流式元数据架构、搜索、血缘、数据负责人体系 | 跨越多存储引擎的统一数据目录 |
| **OpenMetadata / Amundsen** | 数据发现与目录 | 轻量易上手、搜索体验好 | 中小团队快速搭建数据目录 |
| **Apache Griffin** | 数据质量度量 | 规则化质量校验（空值率、重复率、波动）、批流支持 | 数仓质量监控 |
| **Great Expectations** | 用代码声明数据质量预期 | Python 生态、与 Airflow/dbt 集成好 | 数据管道中的质量断言（类似数据的单元测试） |
| **Apache Ranger** | 大数据组件统一权限管控 | 集中式策略管理（Hive/HDFS/Kafka 等）、行级/列级权限、审计日志 | 多组件平台的统一鉴权与合规审计 |
| **Kerberos** | Hadoop 集群身份认证 | 票据机制，解决「你是谁」 | 大数据集群安全基线 |
| **主数据管理（MDM）平台** | 核心实体（客户/商品/组织）多系统口径不一 | 唯一数据源、分发同步 | 集团型企业多系统数据统一 |

### 治理落地要点

1. 治理不是买一个平台，而是「组织规范 + 流程 + 工具」三者结合，先定标准再上工具。
2. 从痛点切入：通常先做**元数据与数据目录**（解决找不到）和**指标口径统一**（解决不敢用）。
3. 质量、血缘、安全跟随数仓建设逐步补齐，避免一开始就大而全。
4. ELT 趋势下，加载进仓后再用 dbt 等工具做转换与测试，治理节点内嵌到数据管道中。
